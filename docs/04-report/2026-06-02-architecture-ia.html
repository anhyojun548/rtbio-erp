1 전체 아키텍처

prototype-as-truth 구조. 화면은 프로토타입 HTML/JS(public/portals/*.html), 백엔드는 Next.js API Route + Server Action, 데이터는 Prisma·PostgreSQL. 브라우저의 모든 동작은 /api/*를 거쳐 실 DB에 반영됩니다.

설계 철학 — 포탈은 “서로 메시지를 주고받지” 않습니다. 모두 같은 DB(단일 진실 원천)를 바라보며, 한 포탈이 상태(status)를 바꾸면 다른 포탈이 그 상태를 읽어 다음 동작을 합니다. 데이터 이동의 매개체는 공유 테이블 · 상태머신 · 스냅샷 · 외래 연결(clientId·billingMonth·salesRepId)입니다.

2 멀티테넌시 · 인증 · 권한

테넌트(고객사)별로 스키마를 분리하고, 역할(role)로 포탈 접근을 통제합니다.

스키마 분리형 멀티테넌시

public(공용: 계정·테넌트·감사) + tenant_{id}(격리: 업무 데이터). 현재 tenant_altibio 1개. 업무 테이블은 스키마 자체가 테넌트 경계, public 테이블은 tenantId 컬럼으로 필터.

인증 — NextAuth (JWT 8h)

이메일+비밀번호(bcrypt). 세션에 {id, role, tenantId, clientId, isTeamAdmin} 주입. 미들웨어가 getToken으로 검증, 미인증 시 /login.

클린 URL

/admin·/qc·… 는 rewrite로 프로토타입 HTML을 서빙(.html 노출 없음). 자산은 절대경로 /portals/....

    5 포탈 연동 구조 — 주문 생애주기

    데이터는 주문(Order) 상태머신을 따라 포탈을 횡단합니다. 한 포탈의 액션이 상태를 바꾸면, 다음 포탈이 그 상태의 주문을 화면에 띄워 다음 액션을 수행합니다. 아래는 발주→출고→정산→경영보고의 한 사이클.

    거래처품질관리 경영지원대표

    →
    발주내역상태/출고 단계 실시간 조회 ←
    품질관리

    →

    마감원장 / 미수금carry+매출−수금=잔액 (billingMonth 링크)
    대표

    핵심 — 각 단계의 “넘김”은 메시지가 아니라 같은 Order 행의 status 변화 + 연관 행 생성입니다. 거래처가 만든 주문을 품질관리가 status로 이어받고, 완료된 주문을 경영지원이 명세서·수금으로 회계화하며, 대표는 그 결과를 읽기만 합니다.

    보조 연동 흐름

    💰 가격 결정 (경영지원 → 발주 시점)

    경영지원·거래처관리에서 할인율(ClientDiscount)·고정가(ClientFixedPrice) 설정 → 발주 생성 시 calculatePriceSnapshot이 고정가 > 할인 > 기본가 우선순위로 OrderItem에 단가를 박제. 이후 제품 가격이 바뀌어도 기존 주문 불변.

    📦 재고 이중관리 (품질관리 중심)

    입고등록(RECEIVE) → 재고조정(반품/폐기/실사/샘플) → 주문 확정 RESERVE(available−) → 출고 SHIP(physical−). 모든 변동은 InventoryLog. physical/available 이중재고 불변식 유지.

    📣 공지 (작성 포탈 → 수신 포탈)

    경영지원/영업/품질/대표가 공지 작성 시 대상 지정(ALL·DEALER·HOSPITAL·SPECIFIC) → 해당 포탈 공지함 + 거래처 포탈 홈/공지로 수신. NoticeRecipient로 특정 거래처 타게팅.

    👤 영업 집계 (영업 ↔ 정산)

    담당자 관리에서 Client.salesRepId/SalesAssignment 배정 → 그 거래처의 매출(Invoice)·미수(ClosingLedger)가 영업 현황·담당자별 매출 보고서로 자동 집계. 학회 방문자(assignedRep) → 영업 이력서.

    🛡 직원·권한 (대표/경영지원 → 각 팀)

    대표·팀관리자 지정에서 isTeamAdmin 부여 → 각 팀의 팀관리자가 직원 관리에서 자기 팀 직원 CRUD. role이 곧 포탈 접근 권한(RBAC).

    📊 위젯·DB (대표·관리자)

    대표 위젯 대시보드 커스터마이징 + windyflo(외부 LLM)가 자연어→WidgetSpec 생성 → DashboardWidget. metaAdmin은 DB 탐색기로 전 테이블 조회 + 설정성 4개 편집.

    7 데이터 무결성을 지키는 핵심 불변식

    포탈을 횡단해도 데이터가 깨지지 않게 하는 규칙들 — 모두 서버 도메인 로직에서 강제.

    ① 가격 스냅샷

    주문 확정 시점의 단가를 OrderItem에 박제(unitPrice·basePriceAtOrder·discountRateAtOrder·fixedPriceAppliedAtOrder). 이후 가격 정책이 바뀌어도 과거 주문·명세서 불변.

    ② 이중 재고 + 로그

    physicalStock(실재고) vs availableStock(가용). 확정=RESERVE(available−), 출고=SHIP(physical−), 취소=RELEASE. 전 변동 InventoryLog. SELECT FOR UPDATE 동시성.

    ③ 배송지 스냅샷

    발주 시 선택한 배송지를 Order.shipTo*에 복사. 이후 거래처 주소가 수정/삭제돼도 과거 주문 배송정보 불변.

    ④ 채번 직렬화

    ORD-/INV-YYYYMMDD-NNN — Postgres advisory lock으로 동시 발번 충돌 방지. 주문/명세서 키공간 분리.

    ⑤ 회계 마감 보호

    ClosingLedger 마감(closedAt) 후 재계산 거부. (clientId, closingMonth) unique + upsert로 직렬화.

    ⑥ 테넌트·행 격리

    업무 데이터는 스키마 격리, 거래처 포탈은 clientId 행-레벨 필터, public 테이블은 tenantId 강제. 민감컬럼(비밀번호)은 select 제외.

    요약 — 5개 포탈은 “하나의 주문이 흘러가는 컨베이어 벨트의 서로 다른 작업대”입니다. 거래처가 올린 주문을 품질관리가 실물로 처리하고, 경영지원이 돈으로 정산하며, 영업이 실적으로 집계하고, 대표가 경영지표로 내려다봅니다. 모든 핸드오프는 공유 DB의 상태 변화로 이뤄지고, 위 6개 불변식이 그 과정에서 데이터가 깨지지 않도록 지킵니다.
