RTBIO ERP 개발을 위한 AI 팀 운영 체계

핵심 3요소

1. CLAUDE.md

프로젝트 도메인 지식을 한 파일에 집약. 모든 에이전트가 자동 참조.

2. 범용 에이전트 풀

32개 역할 기반 서브에이전트 + Team/Autopilot/Ralph 오케스트레이션.

3. 커스텀 도메인 에이전트 3개

schema-designer, pricing-specialist, inventory-specialist. RTBIO 고유 규칙 전담.

3계층 구조

# 호출 흐름 [Claude Code 앱] ↓ 사용자 입력 /team 4:executor "주문 API 구현" [CLAUDE.md] # 자동 로드 — 도메인 규칙 주입 ↓ [오케스트레이션 플러그인] # /team → plan, prd, exec, verify, fix 파이프라인 ↓ [서브에이전트 선택] ├── 범용 32개 중 역할 매칭 (builder, tester 등) └── 커스텀 3개 (schema/pricing/inventory-specialist) ↓ [도구 실행 (Read/Write/Edit/Bash)] ↓ [결과 메인 에이전트에 요약 반환]

에이전트 구성

범용 32개 + 커스텀 3개의 역할 분배

범용 에이전트 (32개)

카테고리: 아키텍처 · 연구 · 디자인 · 테스팅 · 데이터 사이언스

자동 모델 라우팅: 간단한 작업은 Haiku, 복잡한 추론은 Opus

사용자가 개별 호출 불필요. /team, /autopilot이 알아서 역할 매칭.

커스텀 도메인 에이전트 (3개 — 추가 금지)

schema-designer

Phase 1

Prisma 스키마 전담

  • 멀티테넌시 (public + tenant_{id})
  • 감사 컬럼 강제
  • CASCADE 정책 주석

pricing-specialist

Phase 4

가격 계산 로직 전담

  • fixedPrices > discounts > basePrice
  • 스냅샷 컬럼 검증
  • 할인 중복 방지

inventory-specialist

Phase 4

재고 로직 전담

  • physical/available 이중 관리
  • 반품 = adjustment + reason
  • InventoryLog 무결성
⚠️ 추가 금지: 4번째 커스텀 에이전트 만들지 말 것. 범용 32개와 역할 중복됨. "보안 검증 에이전트 필요"? → security 에이전트 있음. "테스트 작성 에이전트 필요"? → tester 있음.

Task 3: 동작 검증 (10분)

롤백 조건

플러그인이 잘 안 맞을 때의 탈출 계획

2주 검토 트리거

아래 중 하나라도 해당 시 플러그인 제거 + 수동 에이전트 방식으로 복귀:

  • ❌ Phase 1 산출물이 도메인 규칙 위반 (가격 스냅샷 누락 등)
  • ❌ 서브에이전트 호출 실패율 > 20%
  • ❌ CLAUDE.md 지식이 에이전트에 전달 안 되는 현상
  • ❌ 토큰 사용량이 과도 (체감상 수동 대비 2배 초과)

롤백 절차

/plugin uninstall oh-my-claudecode # .omc/ 디렉토리 제거 # CLAUDE.md 보존 # .claude/agents/ 의 커스텀 3개는 그대로 활용 # 필요 시 builder/reviewer 에이전트 추가 작성

부분 롤백

일부 기능만 사용

플러그인 유지하되, 특정 명령어만 수동 방식으로 대체. 예: /team 안 쓰고 /autopilot만 사용.

버전 고정

v4.4에서 Codex/Gemini MCP 제거한 전례. 메이저 업데이트 전 특정 버전 pin.

안심 포인트: 롤백해도 CLAUDE.md와 커스텀 3에이전트는 보존됨. 제로부터 다시 시작하는 게 아님.

운영 규칙

하네스를 효과적으로 굴리기 위한 4대 원칙

1. CLAUDE.md = 단일 진실 원천

  • 가격/재고/반품/테넌시 규칙은 CLAUDE.md에만 기록
  • 프롬프트로 반복 주입 금지
  • 규칙 변경 = CLAUDE.md만 수정 (자동 반영)

2. 명령어 적재적소

  • 병렬 독립 → /ultrawork, /team N
  • 단일 일관성 → /autopilot
  • 테스트 존재 → /ralph
  • 중대 결정 → /ccg, /ask

3. 매 Phase 종료 시 /learner

  • 세션 패턴 자동 추출
  • .omc/skills/에 스킬 축적
  • 다음 Phase에서 자동 활용

4. 커스텀 에이전트 최소화

  • 3개(schema/pricing/inventory)로 고정
  • 추가 유혹 참기 — 범용 32개에 대체 있음
  • 중복은 혼란만 만듦

주의사항

하네스 함정과 성공 포인트

⚠️ 하지 말아야 할 것

  • CLAUDE.md 없이 플러그인 설치 — 도메인 지식 없으면 범용 에이전트는 아무것도 모름
  • 커스텀 에이전트 남발 — 3개 이상 추가 금지. 범용 32개와 역할 충돌
  • omc 터미널 CLI 사용 시도 — Windows 미지원(tmux 의존). /명령어 (in-session)만 사용
  • 메이저 버전업 즉시 적용 — v4.4에서 MCP 서버 제거한 전례. 공지 확인 후 신중히
  • CLAUDE.md 비대화 — 60줄 초과 시 토큰 오버헤드. 세부사항은 별도 문서로 분리

✅ 성공 포인트

  • CLAUDE.md 간결 유지 (ETH 연구: 60줄 이내 권장. 긴 파일은 +14~22% 토큰 오버헤드)
  • /learner 적극 활용 — 세션마다 학습 누적
  • /ccg 중대 결정에만 — 3-model이라 비용 있음. 남발 금지
  • Phase 종료 시 회고 — 무엇이 잘 되고 무엇이 안 됐는지 문서화
  • 커스텀 에이전트 지식 CLAUDE.md와 중복 허용 — 방어적 중복은 안전
