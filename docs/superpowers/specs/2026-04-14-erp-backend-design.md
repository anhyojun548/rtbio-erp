# RTBIO ERP — 백엔드 구현 설계서

**작성일**: 2026-04-14
**대상**: 경영지원팀 / 품질관리팀 / 영업팀 / 거래처 포털
**전제**: 프론트 프로토타입(prototype/*.html) 완성, 이제 백엔드 연동 단계

---

## 1. 비즈니스 규칙 (확정)

### 1.1 가격 계산 — 주문 시점 스냅샷
- **우선순위**: 고정단가(`fixedPrices`) > 카테고리 할인율(`discounts[cat]`) > 기본가(`basePrice`)
- **주문 시점에 확정 단가를 DB에 저장** (할인율 변경이 과거 주문에 영향 주지 않음)
- 주문 레코드 저장 필드:
  - `unitPrice` — 실제 적용된 단가
  - `basePriceAtOrder` — 주문 당시 기본가
  - `discountType` — `'fixed'` | `'category'` | `null`
  - `discountPct` — 할인율 (%)
- **거래처별 할인율 엑셀**은 현재 미공유 → 받는 대로 DB에 주입 (스키마는 준비)
- **단가 수동 조정**: 미정 (향후 결정 시 `manualPrice`, `manualReason` 필드 추가 예정)

### 1.2 재고 — 가용재고 / 실재고 분리
- **실재고 (`physicalStock`)**: 창고에 물리적으로 있는 수량
- **가용재고 (`availableStock`)**: `physicalStock` − (확정됐지만 아직 출고 안 된 수량)
- 재고 변동 시점:
  - **발주확정 시**: 가용재고 차감 (예약)
  - **출고완료 시**: 실재고 차감 (확정)
  - **입고완료 시**: 실재고 + 가용재고 모두 증가
  - **재고조정 시**: 실재고 조정, 가용재고 자동 재계산

### 1.3 정산 주기 — 기존 확정 사항
- 거래처별 `closingPeriod` (예: "1일~25일", "전달26일~당월25일")
- 이미 정해진 사항이므로 현 구조 유지

### 1.4 반품 / 교환 — 단순화 (재고조정 통합)
- **별도 메뉴 없음**. 반품 업무는 QC팀의 **재고조정** 페이지에서 통합 처리
- **처리 방식**:
  - 직원이 재고조정 입력 그리드에서 사유 "반품" 선택
  - 비고 필드에 거래처/사유 자유 기록 (예: "메디팜 의료기 반품 - 사이즈 M 2개")
  - 조정 이력에 로그 자동 누적
- **환불 처리**: 시스템 외부에서 처리 (수동 계좌이체)
- **QC 검수**: 별도 워크플로우 없음 (직원 판단으로 재고조정 시 처리)
- **교환**: 별도 기능 없음. 내부적으로 반품 처리 + 신규 출고 분리

### 1.5 세금계산서 — 미정
- 사업자 등록 전까지 기능 보류 (사이드바 비활성화 상태로 유지)

## 2. 프론트 변경 사항

### 2.1 기존 화면 수정

#### [QC팀] 재고현황 (`page-inventory`)
- **변경**: 재고 컬럼을 **실재고 / 가용재고 / 예약** 3개로 분리 표시
- 기존 "현재고" 1개 컬럼 → 3개 컬럼으로 확장
- 상태 뱃지(정상/주의/부족)는 **가용재고 기준**으로 계산

#### [QC팀] 재고조정 (`page-adjustment`) — 반품 업무 통합
- **사유 옵션 추가**: `실사조정 / 반품 / 파손 / 분실 / 기타`
- **비고 컬럼 추가**: 자유 입력 필드 (거래처명, 세부 사유 등 기록)
- 조정 이력 테이블에도 비고 표시
- 반품 케이스도 이 화면에서 처리 (별도 메뉴 없음)

#### [거래처 포털] 발주폼
- **변경**: 제품 선택 시 표시 재고를 **가용재고** 기준으로 변경
- (현재는 `stock` 필드 그대로 사용 → 백엔드 연동 시 `availableStock`으로 전환)

#### [경영지원팀] 거래처관리 — 할인율 입력
- **현 상태**: 이미 카테고리별 할인율 입력 필드 존재 (무릎/상지/하지/스프린트)
- **변경**: 없음. 엑셀 데이터 받으면 데이터만 주입

### 2.2 기존 화면 유지

- 세금계산서: 미정 상태로 유지 (회색 비활성화)
- 단가 수동 조정: 미구현 (향후 추가)

### 2.3 추가하지 않기로 한 화면

- ~~경영지원팀 반품관리~~ → 재고조정으로 통합
- ~~QC팀 반품검수~~ → 재고조정으로 통합

## 3. 백엔드 구현 범위

### 3.1 핵심 도메인
1. **주문 (Order)** — 스냅샷 가격 저장, 상태 머신 (접수→확정→출고중→완료)
2. **재고 (Inventory)** — 실재고/가용재고 이중 관리, 트랜잭션 로그
3. **거래처 (Client)** — 할인율/고정가 관리
4. **반품 (Return)** — 등록 → 검수 → 환불 3단계 워크플로우
5. **거래명세서 / 원장** — 거래처별 closingPeriod 기반 월마감
6. **수금 (Payment)** — 미수금 대조

### 3.2 기술 스택 (기존 개발계획서 유지)
- Next.js 14 + TypeScript
- PostgreSQL + Prisma
- NextAuth.js

### 3.3 구현 우선순위
1. **STEP 1**: 마스터 데이터 (제품, 거래처, 재고) + 인증
2. **STEP 2**: 주문 플로우 (스냅샷 가격 포함) + 재고 차감/복원
3. **STEP 3**: 거래명세서 / 원장 / 수금
4. **STEP 4**: 재고조정 + 조정 이력 (반품 업무 포함)
5. **STEP 5**: 대시보드 통계, 알림, 마감

## 4. 스키마 주요 변경 사항 (기존 개발계획서 대비)

### 기존 `Order` 테이블 → 스냅샷 필드 추가
```
- unitPrice          Decimal  (실제 적용 단가)
- basePriceAtOrder   Decimal  (당시 기본가)
- discountType       String?  ('fixed' | 'category' | null)
- discountPct        Decimal  (할인율 %)
```

### 기존 `Product` 테이블 → 재고 분리
```
- physicalStock      Int      (실재고)
- availableStock     Int      (가용재고, 계산 컬럼 or 캐시)
```

### 신규 `InventoryAdjustment` 테이블 (재고조정 이력)
```
- id, productId, size
- systemStock    Int     (조정 전 재고)
- actualStock    Int     (조정 후 재고)
- adjQty         Int     (차이, +/-)
- reason         String  ('실사조정' | '반품' | '파손' | '분실' | '기타')
- note           String? (자유 입력 비고 — 거래처명/사유 등)
- staffId        String  (처리자)
- createdAt      DateTime
```

### 신규 `InventoryLog` 테이블 (재고 변동 통합 로그)
```
- id, productId, type ('in' | 'out' | 'reserve' | 'release' | 'adjust')
- qty (+/-)
- refId (주문ID / 입고ID / 재고조정ID)
- beforeStock, afterStock
- createdBy, createdAt
```

> 별도 `Return` / `ReturnInspection` 테이블 없음. 반품 업무는 `InventoryAdjustment` 테이블의 `reason='반품'` + `note` 필드로 처리.

## 5. 미결정 사항 (향후 논의)

1. **단가 수동 조정** — 특별 할인가 적용 필요 시 정책 확정 후 구현
2. **세금계산서** — 사업자 등록 완료 시 전자세금계산서 연동 (팝빌 등)
3. **반품 환불 프로세스** — 현재는 시스템 외부(수동 계좌이체) 처리. 추후 환불 기록 추적 필요 시 별도 모듈화 검토
