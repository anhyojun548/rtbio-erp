1. API 공통 규약

1.1 표준 응답

// 성공
{ "ok": true, "data": { ... } }

// 실패
{
  "ok": false,
  "error": {
    "code": "VALIDATION_FAILED",    // 에러 코드 (열거형)
    "message": "수량은 1 이상이어야 합니다",
    "fields": { "qty": "positive number required" }
  }
}

1.2 인증 (NextAuth JWT 쿠키)

  • 모든 /api/* 요청은 next-auth.session-token 쿠키 필요 (/api/auth/*, /api/public/* 제외)
  • 세션에서 tenantId, userId, role을 추출해 Prisma 쿼리에 강제 주입
  • RBAC: 데코레이터 requireRole('ADMIN'|'QC'|'EXEC'|'CLIENT')

1.3 에러 코드 (표준)

코드HTTP의미
UNAUTHENTICATED401세션 없음 / 만료
FORBIDDEN403권한 부족
NOT_FOUND404리소스 없음
VALIDATION_FAILED422Zod 검증 실패
BUSINESS_RULE_VIOLATED409재고 부족 등 비즈니스 규칙 위반
TENANT_MISMATCH403다른 테넌트 리소스 접근 시도
RATE_LIMIT_EXCEEDED429호출 한도 초과

2. 내부 REST API 카탈로그

2.1 인증

POST/api/auth/callback/credentials 구현 예정
이메일 + 비밀번호 로그인 (NextAuth)
Body: { email, password, tenantSlug }
POST/api/auth/signout
세션 종료

2.2 Orders (발주 관리)

GET/api/orders
발주 목록. ?status=&dateFrom=&dateTo=&clientId=&staffId=
POST/api/orders
신규 발주. 재고 예약 + 가격 스냅샷 + 칸반 카드 생성
Body: {
  "clientId": "c1",
  "items": [{ "productId":"P001", "size":"M", "qty":10 }],
  "note": "월말 결제"
}
→ 201 { "orderId":"ORD-260417-001", "totalAmount":500000 }
PATCH/api/orders/[id]
상태 변경 (접수 → 확정/취소). 확정된 주문의 unitPrice는 불변.
PATCH/api/kanban/move
QC 칸반 단계 이동. done 이동 시 physicalStock 차감

2.3 Invoices / TaxInvoices

POST/api/invoices
거래명세서 발급 (Order 기반). invoiceReady=true 인 주문만 가능
GET/api/invoices/[id]/pdf
PDF 다운로드. Blob 캐시 우선, 없으면 @react-pdf/renderer로 생성
POST/api/tax-invoices 팝빌 연동
세금계산서 발행. supply × 0.1 = VAT 자동 계산, 국세청 승인 후 이메일 발송
PATCH/api/tax-invoices/[id]/cancel
세금계산서 취소 (국세청 승인 취소 API 호출)

2.4 Inventory

GET/api/inventory
재고 현황 (physicalStock, availableStock, safetyStock)
POST/api/inventory/adjust
재고 조정 (반품/실사/파손). InventoryAdjustment + InventoryLog 기록
DELETE/api/products/[id]
제품 삭제 (soft delete: status='DISCONTINUED'). OrderItem 참조 있으면 RESTRICT

2.5 Payments

POST/api/payments
은행 입금 수기 등록. 상태 자동 판정 (COMPLETE/PARTIAL/UNPAID)
GET/api/payments/receivables
거래처별 미수금 Aging (0-30 / 31-60 / 61-90 / 90+)

2.6 Clients / Reports

GET/api/clients
거래처 목록. ?type=대리점|병원|기타&staffId=&search=
POST/api/clients
거래처 등록. fixedPrices, discounts 포함. loginId/pw 자동 생성
GET/api/reports/client-ledger
거래처원장. 매출/입금/잔액, 담당자 필터
GET/api/reports/journal
매입매출장. 담당자/기간/거래처 복합 필터

2.7 Dashboard Widgets

GET/api/widgets/[preset]
위젯 프리셋 데이터. ?range=today|7d|30d|month|custom&from=&to=
지원 preset: monthly_sales, today_orders, total_ar, active_clients, weekly_sales, client_share, today_order_list, product_mix, monthly_trend, sales_target, low_stock, ar_clients, client_reps

2.8 Conferences (학회/방명록)

GET/api/conferences
학회 목록 + 집계 캐시 (visitors/contacted/newClients)
POST/api/conferences/[id]/visitors
방명록 개별 등록 (이름/연락처/소속/담당자/상태/비고)

2.9 Settings (QC 포털 설정)

GET/api/settings/qc
업무시간, 택배 마감, 재고 임계치
PUT/api/settings/qc
설정 저장 (ADMIN/QC만)

3. 외부 서비스 연동

📧 Daum SMTP (Nodemailer) 확정

  • 호스트: smtp.daum.net:465 (SSL)
  • 계정: 알티바이오 기존 daum.net 이메일 (운영 시 전달)
  • 용도: 거래명세서/세금계산서 PDF 첨부, 미수금 연체 리마인드, 신규 계정 비밀번호 통지
  • 환경변수: SMTP_HOST, SMTP_PORT, SMTP_USER, SMTP_PASS, SMTP_FROM
  • 월 1,000건 초과 시 Azure Communication Services Email / SendGrid로 전환 (Phase 7 이후)
⚠️ Daum SMTP는 일별 발송 한도가 있으므로 배치 발송 시 큐(Bull) + 지수백오프 필수.

🧾 팝빌 (Popbill) API — 전자세금계산서 Phase 4 도입

  • 엔드포인트: https://popbill.linkhub.co.kr
  • 핵심 API: Taxinvoice_RegistIssue(), Taxinvoice_Cancel(), Taxinvoice_GetPDF()
  • 국세청 전송 자동화 (공인인증서 관리 위임)
  • 환경변수: POPBILL_LINK_ID, POPBILL_SECRET_KEY, POPBILL_CORP_NUM
  • 테스트/운영 모드 분리 (IsTest 플래그)

🏦 은행 입금 조회 (수기 → 향후 Open API) Phase 5+

  • 1단계 (MVP): 경영지원이 admin-portal에서 수기 입력 (현재 프로토타입)
  • 2단계: 농협 OpenAPI / 기업은행 OpenAPI 연동 — 입금 내역 자동 매칭
  • 매칭 키: 입금자명 + 금액 + 날짜 → Client.payerAliases 배열로 다중 별칭 대응

7. 데이터 이관 계획 (얼마에요 → RTBIO ERP)

알티바이오 기존 시스템("얼마에요")에서 약 3년치 데이터를 RTBIO ERP로 일괄 이관합니다. Phase 1 DB 스키마 확정 이후 이관 스크립트를 작성하며, 이관 기간 중 기존 시스템을 병행 운영합니다.

7.1 이관 대상 테이블

테이블도메인비고
clients거래처fixedPrices, discounts 포함
products제품기준단가, 사이즈별 규격
orders주문이력스냅샷 컬럼(unitPrice, basePriceAtOrder) 복원
inventory재고현황physicalStock / availableStock 초기값
payments수금이력입금일, 금액, 잔액 매칭

7.2 이관 제외 항목

  • 세금계산서 원본 — 국세청 홈택스 조회로 대체 (팝빌 연동 이후 재발행 없음)
  • 기타 파일 첨부 (사진, 스캔본 등) — 필요 시 수동 업로드

7.3 이관 방식 및 절차

엑셀 Export → 정제 스크립트 (Node.js/Python) → Prisma Seed → 검증
  1. 기존 시스템에서 테이블별 엑셀 Export
  2. 데이터 정제 스크립트 작성 — 인코딩 정규화, 필드 매핑, 중복 제거
  3. 테넌트 스키마 대상 prisma db seed 실행 (트랜잭션 단위)
  4. 레코드 수 대조 및 샘플 검증 (거래처 10건, 주문 50건 이상 수동 확인)

7.4 일정 및 주의사항

단계시점예상 소요
DB 스키마 확정Phase 1 착수 후—
이관 스크립트 작성 및 테스트스키마 확정 직후1~2일
스테이징 환경 이관 검증스크립트 완료 후1일
운영 환경 Cut-over별도 협의1~2일 (병행 운영 포함)
⚠️ Cut-over 날짜는 알티바이오와 별도 협의 필요. 이관 완료 전까지 기존 시스템("얼마에요") 병행 운영 필수 — 데이터 유실 방지를 위해 단방향 이관(기존→신규)만 허용하며, 신규 ERP 데이터의 역방향 동기화는 지원하지 않음.
