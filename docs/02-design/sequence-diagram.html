1. 로그인 (NextAuth.js + 서브도메인 테넌트)

사용자가 {tenant}.rtbio-erp.com에 접근하면 미들웨어가 테넌트를 식별하고, NextAuth가 bcrypt로 패스워드를 검증한다.
sequenceDiagram
  autonumber
  participant U as 사용자
  participant B as Browser
  participant M as Middleware
(Next.js)
  participant A as NextAuth API
  participant P as PostgreSQL
(public.User)
  participant S as Session

  U->>B: {tenant}.rtbio-erp.com/login
  B->>M: GET /login
  M->>M: 서브도메인 파싱 → tenantId
  M-->>B: 로그인 페이지 (tenant context 주입)
  U->>B: email, password 입력
  B->>A: POST /api/auth/callback/credentials
  A->>P: User.findUnique({email, tenantId})
  P-->>A: user + hashedPassword
  A->>A: bcrypt.compare(pw, hashed)
  alt 검증 성공
    A->>S: JWT 발급 (tenantId, role, userId)
    S-->>B: Set-Cookie next-auth.session-token
    B->>U: /dashboard 리다이렉트
  else 실패
    A-->>B: 401
  end

2. 거래처 발주 → QC 칸반 반영

client-portal 발주 → 재고 예약(availableStock 차감) → QC 칸반에 waiting 카드 생성. 모든 변경은 InventoryLog에 기록.
sequenceDiagram
  autonumber
  participant C as Client Portal
  participant API as /api/orders
  participant ZOD as Zod 검증
  participant DB as Prisma / tenant_{id}
  participant K as QC 칸반 WebSocket

  C->>API: POST /api/orders {items[], note}
  API->>ZOD: OrderCreateSchema.parse(body)
  ZOD-->>API: validated
  API->>DB: BEGIN TRANSACTION
  API->>DB: Product.findMany({ids})
  API->>DB: 가격 계산 (fixedPrices > discounts > basePrice)
  API->>DB: Order.create({unitPrice, basePriceAtOrder})
  API->>DB: Inventory.update availableStock -= qty
  API->>DB: InventoryLog.create(reason='주문예약')
  API->>DB: AuditLog.create(action='ORDER_CREATE')
  API->>DB: COMMIT
  API->>K: publish('kanban.update', {orderId, stage:'waiting'})
  K-->>K: QC 포털 실시간 리렌더
  API-->>C: 201 { orderId, totalAmount }

3. QC 칸반 단계 이동 (drag & drop)

카드를 다음 단계로 드래그하면 stage 필드만 갱신. 출고완료(done)로 이동 시 physicalStock 차감과 거래명세서 발급 가능 플래그가 활성화된다.
sequenceDiagram
  autonumber
  participant QC as QC Portal
  participant API as /api/kanban/move
  participant DB as Prisma
  participant INV as /api/invoice
  participant ADM as Admin Portal

  QC->>API: PATCH /api/kanban/move {orderId, stage:'done'}
  API->>DB: Order.update(stage:'done', doneAt:NOW())
  alt stage == 'done'
    API->>DB: Inventory.update physicalStock -= qty
    API->>DB: InventoryLog.create(reason='출고')
    API->>DB: Order.invoiceReady = true
  end
  API-->>QC: 200 ok
  Note over QC: 카드에 📄 버튼 활성화

  ADM->>INV: POST /api/invoice { orderId }
  INV->>DB: Invoice.create(...)
  INV->>DB: Order.invoiceIssued = true
  INV-->>ADM: 200 { invoiceId, pdfUrl }

4. 거래명세서 PDF 생성 (@react-pdf/renderer)

PDF는 실시간 렌더(다운로드 요청 시 생성). Blob Storage에 캐시 저장.
sequenceDiagram
  autonumber
  participant U as 사용자
  participant API as /api/invoice/[id]/pdf
  participant RP as @react-pdf/renderer
  participant BS as Azure Blob Storage
  participant DB as PostgreSQL

  U->>API: GET /api/invoice/{id}/pdf
  API->>BS: blob.exists(invoice_{id}.pdf)
  alt 캐시 존재
    BS-->>API: URL
    API-->>U: 302 → Blob URL (SAS)
  else 신규 생성
    API->>DB: Invoice + Order + Items + Client 조회
    API->>RP: renderToStream(InvoicePDF props)
    RP-->>API: PDF Buffer
    API->>BS: blob.uploadData(buffer)
    BS-->>API: blob URL
    API-->>U: 200 PDF binary + Cache-Control
  end

5. 세금계산서 발행 (admin-portal → 국세청 연계 대비)

⚠️ 현재 보류: 사업자 등록 전까지 기능 비활성화 (백엔드설계서 §1.5). Phase 4에서 팝빌 API 연동 예정 (bizno, approvalNumber 반환).
sequenceDiagram
  autonumber

6. 입금 수기 등록 → 미수금 자동 반영

admin-portal 수금 페이지에서 은행 입금 내역을 수기로 입력 → 입금액 vs 예상금액 비교 → 상태 자동 판정(입금완료/일부입금/미입금).
sequenceDiagram
  autonumber
  participant A as Admin Portal
  participant API as /api/payments
  participant DB as PostgreSQL
  participant MAIL as Nodemailer

  A->>API: POST /api/payments {clientId, amount, date, bank, payer}
  API->>DB: Receivable.findOne(clientId, openBalance)
  API->>API: 상태 판정 (amount vs expected)
  alt amount == expected
    API->>DB: Payment.create(status:'COMPLETE')
    API->>DB: Receivable.amount = 0
  else amount < expected
    API->>DB: Payment.create(status:'PARTIAL')
    API->>DB: Receivable.amount = expected - amount
  end
  API->>DB: AuditLog.create(action='PAYMENT_REG')
  API-->>A: 200 { status, remaining }

  Note over A,MAIL: 연체 30일 경과 시 배치로 리마인드 이메일
  API->>MAIL: sendMail(client, overdueTemplate)

7. 대시보드 전역 기간 변경 → 위젯 일괄 리렌더

C-1 요구사항: 전역 기간 변경 시 모든 위젯의 개별 override가 리셋되고 리렌더.
sequenceDiagram
  autonumber
  participant U as 사용자
  participant D as Dashboard UI
  participant LS as localStorage
  participant API as /api/widget/[preset]
  participant DB as PostgreSQL

  U->>D: 전역 기간 select change (30d → month)
  D->>D: 어떤 위젯에 override 있는지 확인
  alt override 존재
    D->>U: confirm('override 리셋됩니다. 계속?')
    U-->>D: yes
  end
  D->>LS: globalDateRange = 'month'
  D->>LS: rtbio_dashboard (override 전부 '')
  loop 모든 위젯
    D->>API: GET /api/widget/{preset}?range=month
    API->>DB: 집계 쿼리
    DB-->>API: rows
    API-->>D: JSON
    D->>D: renderWidgetContent(...)
  end
