Phase 4 · 알림 시스템 규모: 중간

주문 확정 · 명세서 발행 · 수금 등록 시 거래처와 담당자에게 이메일/인앱 알림을 보냅니다. Daum SMTP 무료 한도(1,000건/월) 내에서 운영 시작, 초과 시 Azure Communication Services 전환.

구현 항목

항목파일/모듈규모수용 조건
스키마 확장 EmailQueue · Notification · NotificationPreference 작음 재시도 횟수 · 다음 발송 시각 컬럼 · 인덱스
Nodemailer 래퍼 lib/email/{send,template}.ts 작음 Daum SMTP SSL · 템플릿 엔진 · 실패 시 큐 재등록
알림 트리거 actions/{order,invoice,payment}.ts 에 send 훅 중간 SUBMIT/CONFIRM/ISSUE/RECORD_PAYMENT 6개 이벤트
워커 잡 scripts/process-email-queue.ts 중간 5분 cron · 지수 백오프 · 3회 실패 시 DEAD
인앱 알림 UI components/NotificationBell.tsx + /api/notifications/sse 중간 벨 아이콘 · 드롭다운 · 읽음 표시 · 폴링 or SSE
알림 환경설정 UI app/(portal)/settings/notifications 작음 사용자별 ON/OFF · 이메일 수신 주기
주의 — 트리거 시점에 직접 보내지 말고 반드시 EmailQueue 에 적재만 하고 워커가 비동기 처리. 트랜잭션 내에서 SMTP 호출하면 롤백 시 불일치 발생.

Phase 5 · 파일 업로드 · PDF 첨부 규모: 작음~중간

판매 계약서(R20)의 PDF 첨부, 제품 이미지, 학회 사진을 Azure Blob Storage 에 업로드. 현재는 URL 필드만 저장 중 — 실제 업로드 파이프라인 미구현.

항목파일/모듈규모수용 조건
Blob 업로드 래퍼 lib/storage/blob.ts · actions/upload.ts 중간 `@azure/storage-blob` · MIME 검증 · 10MB 제한
계약 PDF 업로드 UI components/ContractForm · PDF 드롭존 작음 드래그앤드롭 · 진행률 · 삭제 확인
제품 이미지 갤러리 ProductForm · 이미지 여러 장 중간 썸네일 생성 · 대표 이미지 지정 · 순서 DnD
SAS 서명 URL api/files/[id]/download/route.ts 작음 RBAC 체크 · 5분 유효 SAS · 감사 로그
파일 감사 로그 AuditLog.action="FILE_DOWNLOAD" 작음 누가 · 언제 · 어떤 파일

Phase 6 · 품질 · QA 규모: 큼

런칭 직전 QA 스프린트. 모바일 반응형이 가장 큰 덩어리 — QC 팀은 창고에서 태블릿/폰으로 쓰므로 필수. E2E 는 회귀 방지용 골격만.

6-1. 모바일 반응형 규모: 중간

터치 우선 UI

  • 최소 터치 타겟 44×44px (WCAG AAA)
  • 사이드바 → 햄버거 메뉴 + 서랍
  • 테이블 → 카드 레이아웃 (sm 브레이크포인트)

포털별 최적화

  • QC — 칸반 수직 스택 · 1-탭 상태 전환
  • 거래처 — 주문 내역 카드 뷰
  • 영업 — 거래처 검색 빠른 접근

6-2. Playwright E2E 규모: 작음

  • 로그인 → 포털 진입 스모크 (포털당 1건)
  • 주문 생성 → SUBMIT → CONFIRM → SHIP → INVOICE → PAYMENT 엔드투엔드 1건
  • CI 통합(GitHub Actions · Playwright Docker)

6-3. 운영 UI 규모: 작음

  • /admin/audit 감사 로그 조회 페이지 (resource · action · user 필터)
  • /account/password 비밀번호 변경 폼 (bcrypt 재해시)
  • Session 만료 안내 토스트 · Refresh 흐름

6-4. 접근성 · 성능 규모: 작음

  • Axe-core 자동 감사 · 키보드 네비게이션 검증
  • Lighthouse 점수 Performance ≥ 85 · A11y ≥ 95 목표
  • 이미지 next/image 적용 · LCP 최적화

Phase 7 · 테넌트 자동 프로비저닝 옵션

5곳 이상 성장 시 활성화. 현재 스키마는 이미 tenant_{id} 로 분리돼 있으나 수동 마이그레이션에 의존. 관리자 화면에서 "새 테넌트 추가" 버튼으로 스키마 복제 · 시드 주입이 자동화돼야 합니다.

flowchart LR A[관리자: 새 테넌트 추가] --> B{중복 체크} B -->|OK| C[public.Tenant INSERT] C --> D["CREATE SCHEMA tenant_new"] D --> E[Prisma 마이그레이션 적용] E --> F[기본 시드 주입
KanbanColumn·Category·기본 관리자] F --> G[Tenant.status=ACTIVE] G --> H[서브도메인 DNS 자동 등록]

주요 결정 사항

  • Prisma multiSchema preview — 런타임 raw SQL 로 CREATE SCHEMA 필요
  • 테넌트 삭제는 soft-delete 만 (스키마 drop 은 수동)
  • 빌링 연동 (Stripe · 월정액) 이 함께 들어올 가능성 높음

권장 진행 순서

일자는 계약 체결 후 확정. 아래는 의존성 기준 진행 순서입니다.

  1. Step 1 · Phase 4 (알림) — 이메일/인앱 알림 연결. 런칭 전 데모 시연 가능해지는 시점.
  2. Step 2 · Phase 5 (파일) + Phase 6-1 (모바일 반응형) — 파일 업로드와 모바일 최적화 병렬 진행. 실 사용자 UAT 개시.
  3. Step 3 · Phase 6-2~4 (E2E · 운영 UI · 성능) — QA 스프린트로 회귀 방지 골격과 운영 도구 확보.
  4. Step 4 · Phase 9 (배포) — Azure 인프라 구성 · 런칭 리허설 · 모니터링 세팅.
  5. Step 5 (옵션) · Phase 7 (자동 프로비저닝) — 2번째 테넌트 온보딩 직전에 활성화.
