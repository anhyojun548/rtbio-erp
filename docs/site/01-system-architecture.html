01. 시스템 아키텍처

5개 포털을 단일 Next.js 앱으로 통합하고, PostgreSQL 스키마 분리로 멀티테넌시를 구현한 구조.

이 문서에서

  1. 개요 — 설계 원칙
  2. 컴포넌트 다이어그램 (논리 구조)
  3. 요청 흐름 (사용자 → 데이터)
  4. 배포 구조 (현재 vs 향후 Azure)
  5. 주요 기술 결정 · 트레이드오프

1. 개요 — 설계 원칙

🏢 멀티테넌시

public 스키마에 공용(User, Tenant, AuditLog), tenant_{id} 스키마에 도메인 데이터(Product, Order, Inventory) 를 물리적으로 분리. 테넌트 격리 + 운영 단순화.

📸 스냅샷 불변성

주문은 체결 시점의 단가·배송지·VAT율을 주문 테이블에 스냅샷. 거래처 가격 · 주소가 변경돼도 과거 주문은 불변.

📦 이중 재고

physicalStock(창고 실수량) + availableStock(예약분 제외 판매가능). CONFIRM 시 availableStock 차감, SHIP 시 physicalStock 차감.

🔒 서버 액션 기반

Next.js 14 App Router + Server Actions 로 REST 없이 타입 안전 RPC. 권한/감사를 한 곳에서 enforce.

3. 요청 흐름 (사용자 → 데이터)

예시: 거래처(CLIENT) 유저가 자기 명세서 PDF 를 다운로드할 때의 흐름.

sequenceDiagram autonumber actor U as 거래처 유저 participant B as Next.js Route
(/client/invoices/[id]/pdf) participant A as NextAuth
Middleware participant G as requireClient() participant S as Server Action
getMyInvoice participant P as @react-pdf participant DB as PostgreSQL U->>B: GET /client/invoices/xxx/pdf B->>A: JWT 쿠키 검증 A-->>B: { userId, role, clientId } B->>G: requireClient() G-->>B: role=CLIENT & clientId 검증 ✅ B->>S: getMyInvoice(id) S->>DB: SELECT invoice
WHERE id=? AND clientId=?
AND status IN (ISSUED,SENT,CANCELLED) DB-->>S: Invoice + items S-->>B: Invoice 데이터 B->>P: renderToStream(InvoicePdfDocument) P-->>B: PDF stream B-->>U: application/pdf
보안 포인트 — DRAFT 명세서는 getMyInvoice 와 PDF 라우트 양쪽에서 필터링. clientId 행-레벨 필터가 항상 enforced 되어 다른 거래처 문서 접근 불가.
