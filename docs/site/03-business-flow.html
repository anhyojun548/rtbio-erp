03. 비즈니스 플로우

핵심 도메인 프로세스 5종. 각 플로우는 실제 코드에 대응 — 상태머신 · 시퀀스 · 파이프라인.

1. 발주 주문 수명주기

주문은 7개 상태로 움직이며, 각 전환마다 라인별 재고 동시성 제어 · 스냅샷 재기록 · 감사 로그가 실행됩니다.

📋 SUBMIT 전환

  • Postgres advisory lock 기반 채번
  • 라인 가격 재스냅샷
  • billingMonth = YYYY-MM

✅ CONFIRM 전환

  • 라인별 SELECT FOR UPDATE
  • availableStock -= qty
  • InventoryLog.RESERVE 기록
  • 재고 부족 시 전체 롤백

🚚 SHIP 전환 (terminal)

  • 라인별 SELECT FOR UPDATE
  • physicalStock -= qty
  • InventoryLog.SHIP 기록
  • Order.status=COMPLETED

3. 이중 재고 차감 시퀀스

창고 실수량(physicalStock)과 예약분 제외 판매가능수량(availableStock)의 2단계 관리. 동시성은 라인별 SELECT FOR UPDATE 로 격리.

WHERE id IN (...)
FOR UPDATE DB-->>TX: 행 잠금 획득 TX->>DB: UPDATE availableStock -= qty DB-->>TX: OK TX->>LOG: INSERT InventoryLog(RESERVE) API-->>U: Order.status = CONFIRMED Note over U,LOG: 2. 불변식 — availableStock ≤ physicalStock 은 항상 유지. 반품(InventoryAdjustment.reason=반품) 은 physicalStock 만 조정. 샘플출고(QC 포털)는 음수 qty 로 차감.

1순위 · ClientFixedPrice

거래처마다 특정 제품+사이즈에 대한 개별 계약가. 카테고리 할인율을 무시하고 우선.

2순위 · ClientDiscount

거래처×카테고리 할인율 (0~1). 예: "스테이플러" 카테고리 15% 할인.

3순위 · basePrice

ProductSize 의 기본가. 거래처 할인 규칙이 없을 때 적용.

5.1 VAT 계산

calcVatTotal(supply, rate) — rate 는 TenantSetting.vat_rate 에서 읽음 (기본 0.10). supplyAmount = Σ(lineTotal), vatAmount = supplyAmount × rate, totalAmount = supplyAmount + vatAmount.

5.2 Payment 소프트 취소

수금 취소는 hard delete 가 아닌 status=PENDING + note 에 [취소] 태그 덧붙이기. BankTransaction.matched 는 별도 unmatchBankTxn 으로 해제. 집계에서는 PENDING 제외.

5.3 원장 동시성

(clientId, closingMonth) unique + upsert 로 여러 재계산이 동시에 실행돼도 직렬화. recomputeLedgerMonth 는 per-client 트랜잭션을 격리 — 1곳 실패가 다른 거래처 집계를 막지 않음.
