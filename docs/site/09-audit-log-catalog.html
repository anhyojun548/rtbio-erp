감사 로그 카탈로그

모든 상태 변경은 AuditLog 테이블에 기록됩니다. 이 문서는 현재까지 정의된 72개 action 코드 의 의미·연관 리소스·metadata 키를 정리한 표준 카탈로그입니다. 규제 대응·문제 추적·사용자 조사용 단일 진실 원천.

정의된 action

72

domain × verb

도메인 그룹

15

CLIENT · ORDER · …

로그 기록 액션

100%

모든 쓰기 = logAudit

보관 주기

∞

소프트 삭제 없음

AuditLog 스키마

model AuditLog {
  id         String   @id @default(cuid())
  tenantId   String
  userId     String?               // 시스템 이벤트는 null
  action     String                // "ORDER_SUBMIT" 등 (본 카탈로그)
  resource   String                // "Order:cuid123" 같이 TYPE:ID 형식
  metadata   Json?                 // action 별 부가 정보
  ipAddress  String?
  userAgent  String?
  createdAt  DateTime @default(now())

  @@index([tenantId, createdAt])
  @@index([tenantId, action])
  @@index([tenantId, resource])
}

명명 규약

  • 형식: DOMAIN_VERB (예: ORDER_SUBMIT) 또는 DOMAIN_SUB_VERB (예: CLIENT_ADDRESS_CREATE)
  • 대문자 + 언더스코어 · 동사는 현재형 · 과거형 금지
  • resource: TYPE:id 형태. TYPE 은 Prisma 모델명 PascalCase
  • metadata: 사람이 읽을 수 있는 요약 + 롤백에 필요한 이전 값

조회 패턴 (향후 UI)

현재 /admin/audit 조회 UI 는 미구현(Phase 6 에 예정). 아래는 SQL/Prisma 쿼리 예시입니다.

특정 주문의 전체 이력

await prisma.auditLog.findMany({
  where: {
    tenantId,
    OR: [
      { resource: `Order:${orderId}` },
      // 관련 invoice, shipment 도 같이
      { action: { startsWith: "SHIPMENT_" } },
    ],
  },
  orderBy: { createdAt: "asc" },
});

사용자별 활동

await prisma.auditLog.findMany({
  where: {
    tenantId,
    userId: userId,
    createdAt: { gte: from, lt: to },
  },
  orderBy: { createdAt: "desc" },
  take: 100,
});

운영 가이드

📋 새 action 추가 시

1. 본 카탈로그에 먼저 추가 → 이름·리소스·metadata 정의
2. 액션 함수에서 logAudit({...}) 호출
3. 스모크 테스트로 로그 기록 검증

✅ 변경 불가 원칙

AuditLog 는 UPDATE/DELETE 금지. 잘못된 로그는 새 action 으로 보정(예: ORDER_SUBMIT_CORRECTED).

⚠️ PII 주의

metadata 에 주민번호·신용카드·계좌번호 기록 금지. 거래처 연락처도 최소화. 감사 로그는 개발자도 조회 가능.

🔒 보관 정책

법정 보관기간 5년 (전자세금계산서 근거). 5년 이전 로그는 cold storage 이관 (Phase 6 이후 고려).
